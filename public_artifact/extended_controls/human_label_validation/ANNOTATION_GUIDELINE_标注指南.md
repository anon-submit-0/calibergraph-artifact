# MetricCaliberBench 人工标注效度研究 · 标注指南（发布版）

## 目的
验证基准金标是否反映真实业务意图（而非编译器作者的自嗨）。你的标注将用于计算标注者间一致性（Cohen's kappa）以及与已发布金标的一致率。**请独立标注，不要与其他标注者讨论，不要查看任何金标文件。**

## 你会拿到
1. `annotation_sheet_v1.csv`：120 条自然语言取数请求（分层抽样自 5 个公开层）。
2. 每层的治理契约材料（只读）：`public_artifact/public_benchmark/<layer>/metric_catalog.jsonl`（指标名/别名/公式/允许维度/是否可答）、`dimension_catalog.jsonl`（维度与层级）、`policy_catalog.jsonl` 或 README 里的拒答政策。

## 标注任务（每条请求 4 个格子）
1. **action**：该请求按治理政策应当回答（answer）还是拒答（refuse）？拒答情形包括：目录中不存在的指标、要原始行/个人敏感信息、SQL/DDL 请求、无法确定口径的含糊请求。
2. **metric_id**：若回答，应选目录中哪个指标（填 metric_id；别名要归一到正式 id）。
3. **dimensions**：应按治理规则输出哪些维度（分号分隔）。**层级规则**：若请求同时提到同一层级路径上的多级（如"按县和城市"），按"最细粒度"政策只保留最细一级——如果你认为业务上应双列保留，请照你的真实判断标注并在第 4 列注明，这正是我们要测的。
4. **不确定**：拿不准就打 `?` 并写一句原因，不要猜。

## 预计用时
约 60–90 分钟／人。请一次性完成避免标准漂移。

## 提交
把你填完的 CSV 重命名为 `annotation_sheet_v1_<你的姓名拼音>.csv` 发回。
